# 卫生健康平台部门分类系统功能与下一步开发报告

生成日期：2026-06-30
系统范围：`chronic-care-platform` 卫生健康信息平台 MVP
报告用途：按部门/角色和主要功能模块，梳理当前已完成能力、下一步可开发能力和优先建设顺序。

## 一、总体判断

当前系统已经从单一慢病管理演示扩展为覆盖“居民服务、医疗协同、医保监管、医政妇幼、质量安全、医院运行、科研治理、平台运维”的综合卫生健康信息平台 MVP。

根据当前发布证据，系统发布报告为 `PASS`，检查项 114/115 通过，剩余 1 项为生产切换阶段的审计留存配置告警；存储快照包含 113 个集合、444 条记录，SQLite 模型包含 19 张表、schema v7。综合驾驶舱跟踪 8 个优先应用，覆盖 348 条来源记录、12 个开放事项、10 个高风险事项、9 条接口轨道和 8 条证据记录。

当前状态可概括为：

- 演示与互联网预览：已具备。
- 受控试点：居民端、互联网护理、陪诊、慢病随访等具备试点基础。
- 正式生产上线：仍需真实身份源、外部接口、生产数据库、审计保全、监控告警、灾备演练、等保密评和现场签字。

## 三、按模块归纳的主要已完成能力

已完成：

- 健康档案、电子病历、检查检验、用药处方、影像资料和附件资料分类展示。
- 家庭成员切换、居民授权共享、访问历史和居民端移动预览。
- 互联网护理申请、机构评估、护士接单、护理记录、质量回访。
- 陪诊预约、服务主体、陪诊人员、合同保险、补贴和风险回访。

下一步：

- 真实居民身份、短信网关、监护关系核验。
- 健康档案和电子病历真实接口接入。
- 授权撤销后端强制拦截。
- 居民端消息推送、小程序/APP 发布策略、服务评价和投诉闭环。

### 2. 医疗协同类

## 四、下一步开发优先级建议

### P0：居民端试点上线底座

优先目标：让居民端形成“真实可试点”的小范围上线能力。

建议开发：

- 实名身份、短信网关、家庭成员/监护人关系核验。
- 健康档案、电子病历、检查检验、用药记录只读接口。
- 授权共享强制拦截、撤销即时生效、访问日志联动。
- 居民服务试点白名单、应急下线开关、隐私协议和服务协议。

### P1：医疗机构与医共体真实联调

优先目标：让居民端和监管端看到真实医疗数据。

建议开发：

- HIS、EMR、LIS、PACS、心电、体检系统接口联调。
- 转诊号源、床位、远程视频、报告回传和下转随访消息。
- 护理服务、陪诊服务和家庭医生服务的机构侧运营闭环。

### P2：医保、证照、统计与监管闭环

优先目标：形成跨部门协同和验收材料。

建议开发：

- 医保核心结算、医保电子凭证、门慢门特、异地转诊规则。
- 电子证照、公安户籍、妇幼、疾控、民政共享。
- 质量安全、药耗监管、医师执业监管的整改闭环和信用联动。

### P3：生产运维与安全合规

优先目标：达到正式生产上线门槛。

建议开发：

- PostgreSQL 或正式生产数据库适配。
- 审计日志保全、SIEM 对接、国密传输、等保密评。
- 生产监控、告警、值班、灾备演练和上线签字。

## 五、建议第一批上线范围

第一批建议上线“居民健康服务试点版”，控制范围如下：

| 功能 | 上线方式 | 说明 |
|---|---|---|
| 健康档案 | 只读 | 展示基本信息、指标、慢病标签、风险提示和来源机构 |
| 电子病历 | 只读 | 展示诊疗摘要、检查检验、用药、出院摘要，不提供诊疗解释 |
| 互联网护理 | 可申请、可追踪 | 居民申请、机构评估、护士派单、护理记录、质量回访 |
| 陪诊服务 | 可预约、可回访 | 预约、服务主体、陪诊人员、合同保险、风险回访 |
| 慢病随访 | 可提醒、可反馈 | 随访计划、居民反馈、家庭医生处理 |
| 授权共享 | 试点启用 | 先用于平台内授权记录和审计，不直接开放第三方自由调阅 |

暂不建议第一批承诺：

- 在线诊疗。
- 处方支付。
- 医保实时结算。
- 正式挂号退号。
- PACS 原图下载。
- 第三方自由调阅居民健康档案。

## 六、结论

系统当前已经完成了“功能骨架、演示闭环、发布证据和多部门模块映射”，具备开展互联网预览和受控试点的基础。下一阶段不宜继续单纯堆叠新页面，而应围绕居民端试点上线，优先补齐真实身份、真实数据接口、授权审计、护理陪诊运营闭环和生产运维合规。

建议建设主线为：

居民端试点上线 → 医疗机构真实接口联调 → 医保/证照/统计跨部门协同 → 生产运维与安全合规 → 正式推广上线。
