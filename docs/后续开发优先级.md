## 一、总体判断

当前系统已经覆盖市级管理、医疗机构、医保、居民、县域医共体和统一运营工作台，并形成慢病、转诊、医保审核、固定取药、统计证照、平台建设治理等演示闭环。

下一阶段不再优先增加展示页面，而是先完成生产底座。排序原则为：安全与数据正确性优先于新功能，真实跨端闭环优先于新增看板，可自动验收优先于人工演示。

## 二、系统现状与主要风险

| 层面 | 已有能力 | 主要风险 |
|---|---|---|
| 业务 | 多角色入口和主要业务链路已覆盖 | 部分流程仍依赖演示状态和人工维护 |
| 权限 | 页面守卫、API 角色校验、居民数据裁剪 | 演示密码、内存会话、缺少字段级授权和撤权复核 |
| 数据 | SQLite 主存储、JSON 静态快照、居民主索引、身份/健康档案/慢病/随访/医保/证照/诊疗/取药/县域业务结构化镜像表、集合级乐观锁、409 冲突提示、集合级保存 API、居民 PATCH、个人健康档案/工作流/证照/医保/取药/慢病计划/县域业务版本化写入、恢复演练、快照数据质量校验、脱敏副本生成和操作手册 | 更多业务级写入入口和生产级恢复指标验收仍需补齐 |
| 接口 | 已有状态、证照、档案和工作流 API | 缺少版本化契约、幂等键、接口模拟、失败重试和生产网关 |
| 工程 | 全量 JavaScript 语法检查 | 没有单元测试、集成测试、端到端测试和 GitHub Actions |
| 安全合规 | 访问日志、安全事件、等保密评信创台账 | 审计尚不可篡改，脱敏、密钥、国密和容灾仍是台账状态 |
| 运维 | 本地服务与 GitHub Pages 静态演示 | 静态站不运行 Node API，尚无后端持续部署和环境分层 |

## 三、开发优先级

### P0：进入真实试运行前必须完成

#### P0-1 自动化测试与持续集成

- 为登录、角色隔离、居民数据裁剪、工作流状态机、证照和档案接口建立测试。
- 增加 API 集成测试和各角色核心旅程端到端测试。
- 建立 GitHub Actions：语法检查、测试、敏感信息扫描、静态站构建与部署检查。
- 验收标准：所有角色越权用例返回 401/403；核心流程自动回归；主分支合并前检查必须通过。

#### P0-2 正式数据模型与迁移体系

- 将集合级 SQLite/JSON 状态拆分为居民、账户、档案、慢病、随访、转诊、医保、证照、审计等正式表。
- 增加主键、外键、唯一约束、索引、乐观锁、事务和迁移版本。
- 建立备份、恢复、脱敏副本和数据质量校验。
- 验收标准：并发写入不丢失；迁移可前进和回滚；完成备份恢复演练；JSON 仅作为脱敏演示快照。

#### P0-3 生产认证、授权和隐私保护

- 接入政务统一身份、机构目录及医生身份源，替换演示密码和内存会话。
- 建立机构、岗位、居民关系、数据范围、字段敏感级别和用途授权模型。
- 支持授权撤销、历史访问复核、敏感字段脱敏和审计保全。
- 验收标准：页面、API、字段三级权限一致；跨机构和跨居民访问自动拒绝并留痕；会话密钥可轮换。

#### P0-4 接口契约与集成网关

- 为 HIS、EMR、LIS、PACS、心电、医保、统计、电子证照建立版本化接口契约和字段映射。
- 增加业务幂等键、签名验签、超时重试、死信补偿、对账和接口运行监控。
- 建立模拟服务与联调测试数据，避免页面直接依赖演示集合。
- 验收标准：至少贯通一所医疗机构的患者、病历、检验和转诊链路；失败可追踪、可重放、可对账。

### P1：形成区域业务闭环

#### P1-1 检查检验互认和共享中心

- 完成影像、心电、检验申请、报告回传、危急值、质控和互认规则。
- 将待互认、退回复核、医保调阅和重复检查提醒接入工作流。
- 验收标准：申请、执行、报告、互认、异常处置全流程有状态、有责任人、有审计。

#### P1-2 统一任务与消息中心

- 汇总随访、转诊、取药、审核、危急值、整改和证照待办。
- 支持站内消息、短信或政务消息渠道、送达回执、升级提醒和免打扰策略。
- 验收标准：任务从产生到完成可追踪；超时自动升级；居民消息必须经过授权与隐私校验。

#### P1-3 数据质量与主索引治理

- 建立重复居民识别、合并拆分、来源可信度、字段完整性、异常值和延迟监控。
- 将统计质控问题、接口差异和机构信用扣分关联到整改工单。
- 验收标准：每条关键数据可追溯来源；主索引冲突可人工复核；质量问题有闭环时限。

#### P1-4 安全合规技术落地

- 把等保、密评、国密改造和信创适配从台账推进到环境、配置和测试证据。
- 实施日志防篡改、密钥管理、数据库加密、漏洞管理和容灾演练。
- 验收标准：四类台账均挂接真实证据；高风险问题清零；恢复时间和恢复点目标通过演练。

### P2：提升治理、科研和居民体验

#### P2-1 信用评价与绩效自动计算

- 将机构信用评分从示例值改为规则版本、数据来源、扣分明细、申诉和公示流程。
- 形成医共体绩效、药耗、人财物和基层履约报表。

#### P2-2 科研数据集与专病库治理

- 建立伦理审批、脱敏、数据集版本、授权、使用审计和成果回流。
- 优先选择高血压、糖尿病等已有数据基础的专病库试点。

#### P2-3 移动端、适老化和无障碍

- 完成大字模式、读屏语义、家属代办、线下帮办、消息触达和弱网体验。
- 按真实老年用户任务完成可用性测试，而不是只验证响应式布局。

## 四、建议实施顺序

1. 第一迭代：测试框架、CI、核心权限回归、API 契约基线。
2. 第二迭代：正式数据库表结构、迁移工具、并发写入和备份恢复。
3. 第三迭代：统一身份、细粒度授权、脱敏与审计保全。
4. 第四迭代：选择一家机构贯通 EMR/LIS/PACS 与转诊、互认闭环。
5. 第五迭代：统一任务消息、数据质量治理、安全合规证据闭环。
6. 第六迭代：信用绩效、科研专病库和移动适老化深化。

## 五、当前发布边界

GitHub Pages 仅发布静态页面与脱敏数据快照，适合方案展示。Node API、SQLite、真实会话和跨端共享写入必须部署到独立后端环境；在 P0 完成前，不应将当前演示站描述为生产系统。

## 六、工期与 Codex 用量安排

以下估算按 Codex 每周 5 天可用、每天 1 至 2 个“实现 + 测试 + 审查”的聚焦工作块计算。Codex 无法读取账户剩余配额，因此以可独立合并的小批次控制用量；若每周只能工作 3 天，整体工期增加约 30% 至 50%。

| 阶段 | 预计工作块 | 代码侧工期 | 含外部协调工期 |
|---|---:|---:|---:|
| P0-1 测试与 CI | 8–12 | 1–2 周 | 1–2 周 |
| P0-2 数据模型与迁移 | 18–30 | 4–6 周 | 5–8 周 |
| P0-3 生产认证与隐私 | 15–25 | 3–5 周 | 5–8 周 |
| P0-4 接口契约与网关 | 20–35 | 4–7 周 | 6–12 周 |
| P1 区域业务闭环 | 35–55 | 7–11 周 | 10–16 周 |
| P2 治理、科研、移动体验 | 25–45 | 5–9 周 | 8–14 周 |

纯代码与模拟接口全部完成约需 18 至 28 周。达到真实试运行标准、包含医疗机构接口联调、身份源、安全测评和恢复演练，预计需要 24 至 36 周；外部系统窗口或测评排期不确定时，按 9 至 12 个月准备更稳妥。

Codex 用量建议前期按 P0 60%、P1 30%、P2 10% 分配。每个批次控制在一个主题、一个回归测试集和一个 PR 内，优先保证测试与权限审查，不并行铺开多个需要大规模上下文的模块。
